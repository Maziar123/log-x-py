Advanced Techniques

Dispatcher Connections

    

The CodeSiteManager.DefaultDestination and T|CodeSiteLogger.Destination properties are used to specify where (and how) CodeSite messages are routed to their final destination after they are received by a CodeSite Dispatcher.  As you can see from the CodeSite Architecture diagram, a CodeSite Dispatcher is always responsible for updating an instance of the Live Viewer, or updating a CodeSite log file.

However, you can also see from the CodeSite Architecture that there are two different ways T|CodeSiteLogger instances can connect to a CodeSite Dispatcher. Specifically, a T|CodeSiteLogger instance can connect to a local Dispatcher using the wm_CopyData window message protocol (the default). Or, a T|CodeSiteLogger instance can connect to a local or remote Dispatcher using the TCP protocol. While a T|CodeSiteDestination defines what happens to a CodeSite message once it reaches a Dispatcher, a Dispatcher Connection defines how the message is transported to the Dispatcher.

Changing the way logger instances connect to the Dispatcher is very easy. The CodeSiteManager defines several methods for configuring the default connection:

    ConnectUsingCopyData;

    ConnectUsingTcp( host, port );

    ConnectUsingTcp( host );

    ConnectUsingTcp;

 

By default, the CodeSiteManager internally calls ConnectUsingCopyData, which is very effective is most situations. However, there are times when the wm_CopyData protocol is not applicable (e.g. Logging from Services, ASP.NET). Plus, wm_CopyData cannot be used to connect to a remote Dispatcher. In these situations, one of the ConnectUsingTcp methods must be used. For example:

    // Connect to local Dispatcher using default port

    CodeSiteManager.ConnectUsingTcp; 

 

    // Connect to the HelpDesk machine on port 4040. 

    // The Dispatcher must be configured to monitor port 4040 for TCP messages.

    CodeSiteManager.ConnectUsingTcp( 'HelpDesk', 4040 );

 

By default all T|CodeSiteLogger instances will connect to the Dispatcher in the manner defined by the CodeSiteManager.  However, if you would like to override the connection mechanism for a particular logger instance, you may do so with the following methods:

    ConnectUsingCopyData;

    ConnectUsingTcp( host, port );

    ConnectUsingTcp( host );

    ConnectUsingTcp;

    UseCodeSiteManagerConnection;

 

As you can see, aside from the UseCodeSiteManagerConnection method, the T|CodeSiteLogger methods above are identical to the CodeSiteManager versions. The UseCodeSiteManagerConnection method removes any existing connection override being used by the logger instance and once again uses the connection that is defined by the CodeSiteManager.
