Sending Custom Data: TCodeSiteFormatter (C++Builder)

    

Note: *Not Available in CodeSite Express

The following is the complete C++Builder source code for the BaseballPlayer sample unit (BaseballPlayer.hpp and BaseballPlayer.cpp), which implements a custom formatter for the TBaseballPlayer record data type.

Let's start with the BaseballPlayer.hpp file:

#ifndef BaseballPlayerHpp

#define BaseballPlayerHpp

 

const csmPlayerSummary = csmUser + 1;

const csmPlayerDetails = csmUser + 2;

 

The first thing that we need to do in our custom unit header file is to define a new CodeSite message type constant for each formatter class that we are going to create.  In this example, two formatter classes will be created.  Therefore, we define two new constants.

If you are creating the formatters for your own personal use or for use within your department or organization, then you should define your constants based on the csmUser constant plus some offset.  The valid offset range is 1 to 32,000.  If you are creating the formatters for public use outside of your department or organization, then you should define your constants based on the csmPublic constant plus some offset.  Please contact Raize Software (support@raize.com) to obtain a valid offset that does not conflict with other public formatters.

struct TBaseballPlayer

{

  UnicodeString LastName;

  UnicodeString FirstName;

  UnicodeString Position;

  Currency Salary;

  int Games;

  int AtBats

  int Hits;

  Single ERA;

  bool AllStar;

};

 

The above structure defines the custom data type that we want to send using CodeSite.  Please note that the custom data type does not have to reside in the same unit as the custom formatters.  It is done so here to simplify the example.  Also, any code that requires this data type will also automatically include the necessary formatters.

class TPlayerSummaryFormatter : public TCodeSiteFormatter

{

public:

  virtual TCodeSiteInspectorType __fastcall InspectorType(void);

  virtual void __fastcall FormatData(void *Data);

  virtual UnicodeString __fastcall TypeName(void);

};

 

 

class TCSPlayerDetailsFormatter : public TCodeSiteFormatter

{

public:

  virtual TCodeSiteInspectorType __fastcall InspectorType(void);

  virtual void __fastcall FormatData(void *Data);

  virtual UnicodeString __fastcall TypeName(void);

};

 

#endif

// end of BaseballPlayer.hpp

 

These are the class declarations for the two custom formatters defined in this unit.  The first formatter class will format a TBaseballPlayer struct into a text format, while the second formatter will format a TBaseballPlayer struct into a name-value grid format.

Next, we'll look at the BaseballPlayer.cpp file:

__fastcall TPlayerSummaryFormatter::InspectorType(void)

{

  return itStockText;

}

 

The first step in implementing a custom formatter is to determine which inspector will be used by the CodeSite Viewer to view the formatted data.  The inspector type will determine which helper methods are used in the FormatData method.  For the player summary formatter, the stock text inspector within the CodeSite Viewer will be used.

void __fastcall TPlayerSummaryFormatter::FormatData(void *Data)

{

  TCSBaseballPlayer *Player = (TCSBaseballPlayer *) Data;

 

  AddLine( Player->FirstName + " " + Player->LastName );

  if ( Player->AllStar )

    AddLine( "All-Star " + Player.Position );

  else

    AddLine( Player->Position );

  AddLine( "" );

  AddLine( "Salary: " + FloatToStrF( Player->Salary, ffCurrency, 18, 2 ) );

  AddLine( "" );

  AddLine( "Games Played: " + IntToStr( Player->Games ) );

 

  if ( Player->Position == "Pitcher" ) 

    AddLine( "ERA: " + FormatFloat( "0.00", Player->ERA ) )

  else

    AddLine( "Batting Avg: " + 

             FormatFloat( "0.000", (float) Player->Hits / Player->AtBats ) );

}

 

The FormatData method is where the real work is performed.  Note that the parameter to FormatData is an untyped variable parameter.  This means that any data type may be passed to this method.  However, because of the registration process that will follow we can safely typecast the passed-in parameter to the type we expect to format.

Fortunately, actually formatting the data is not very complicated because helper methods are defined in the TCodeSiteFormatter base class. Since the InspectorType method above specified the itStockText inspector type, the AddLine helper method is used to add lines to the formatted string list.

UnicodeString __fastcall TPlayerSummaryFormatter::TypeName(void)

{

  return "TCSBaseballPlayer";

}

 

Overriding the TypeName is optional, but recommended.  The message details panel in the CodeSite Viewer will show the TypeName of the data type associated with the message.

For the player details formatter, the stock name-value grid inspector within the CodeSite Viewer will be used.

void __fastcall TCSPlayerDetailsFormatter::FormatData(void *Data)

{

  TCSBaseballPlayer *Player = (TCSBaseballPlayer *) Data;

 

  AddNameValuePair( "LastName", Player->LastName );

  AddNameValuePair( "FirstName", Player->FirstName );

  AddNameValuePair( "Position", Player->Address );

  AddNameValuePair( "Salary", FloatToStrF( Player->Salary, ffCurrency, 18, 2 ) );

  AddNameValuePair( "Games", Player->Games );

  AddNameValuePair( "AtBats", Player->AtBats );

  AddNameValuePair( "Hits", Player->Hits );

  AddNameValuePair( "ERA", Player->ERA );

  AddNameValuePair( "AllStar", Player->AllStar );

}

 

To format you custom data for the name-value grid inspector, use the AddNameValuePair method.  The AddNameValuePair method is overloaded with several variations to minimize the need for converting your data to a string format.

In order for the SendCustomData method to access to the custom formatter classes, we must first register them with the CodeSiteManager by calling the CodeSiteManager.RegisterCustomFormatter method.

CodeSiteManager->RegisterCustomFormatter( csmPlayerSummary, __classid( TCSPlayerSummaryFormatter ) );

 

CodeSiteManager->RegisterCustomFormatter( csmPlayerDetails, __classid( TCSPlayerDetailsFormatter ) );

 

Note that the corresponding message type is passed along with the class reference of the appropriate formatter.  The CodeSiteManager maintains a list of all registered formatters indexed by the message type.  When the CodeSiteObject.SendCustomData method is called, the message type passed determines the formatter used to format the data.
